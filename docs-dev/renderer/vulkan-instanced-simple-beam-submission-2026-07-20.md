# Native Vulkan Instanced Simple-Beam Submission

Date: 2026-07-20

Task ID: `FR-01-T14`

Status: complete for the bounded simple/lightning beam submission slice.

## Outcome

The native Vulkan renderer now submits ordinary camera-facing beams and each
non-polygonal lightning segment as one 40-byte `{ start, end, right, color }`
instance by default. The CPU still selects the exact legacy facing vector from
the beam direction and view origin; the Vulkan vertex shader expands the
original six-vertex ordering `(0, 2, 3), (0, 1, 2)` from `gl_VertexIndex`.
This preserves the original world positions, UVs, fullbright/no-shadow/no-dlight
flags, fog handling, descriptor selection, alpha blending, and depth-hack
behavior through the existing native entity fragment and blend paths.

The compact beam instances occupy an independently grown region in the
existing per-frame Vulkan instance arena and join its single multi-region copy
and transfer-to-vertex-input barrier. Adjacent compatible simple beam segments
coalesce without changing their draw order. Polygonal beams retain their
existing native CPU-expanded Vulkan geometry path because their radial surface
topology is distinct.

`vk_beam_gpu_instancing` is an archived Vulkan-only, default-on (`1`) cvar.
Setting it to `0`, or a GPU-pipeline creation failure, retains the prior
CPU-expanded native Vulkan path for focused diagnostics. No Vulkan path routes
to OpenGL.

Each simple segment previously staged six 48-byte dynamic vertices (288 bytes);
its compact instance is 40 bytes, an 86.1% reduction in segment-record upload
volume before any driver-side timing claim.
