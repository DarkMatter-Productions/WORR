# Native Vulkan Instanced Particle Submission

Date: 2026-07-20

Task ID: `FR-01-T14`

Status: complete for the bounded billboard-particle submission slice.

## Outcome

`VK_Entity_AddParticles` now uses a native Vulkan instanced billboard path by
default. Instead of CPU-expanding three `vk_vertex_t` records per particle,
the frame-local entity instance arena carries one compact 20-byte
`{ origin.xyz, scale, color }` record. The Vulkan vertex shader reconstructs
the original three camera-facing triangle corners and their legacy
`1.70710678` texture coordinates from `gl_VertexIndex`.

The shader derives the same right/up view axes from the existing entity push
matrix, preserves the original distance-scaled particle size and colour/alpha
calculation, and emits the unchanged fullbright/no-shadow/no-dlight flags.
The established entity fragment shader, texture shapes, global/height fog,
and alpha/additive blend pipelines therefore remain shared native Vulkan
behavior. Particle batches coalesce only adjacent particles with the same
descriptor, blend style, and submission phase, retaining draw order.

The compact records occupy an independently grown, aligned `particle` region
inside the existing per-frame model instance arena. The live bytes join its
single multi-region transfer and vertex-input barrier. No OpenGL renderer path
is included or invoked.

`vk_particle_gpu_instancing` is an archived, default-on (`1`) Vulkan-only
cvar. Setting it to `0` retains the prior CPU-expanded geometry path, still
submitted natively through Vulkan, for focused A/B investigation or a driver
workaround. Pipeline-creation failure similarly preserves that Vulkan fallback.

For the 8,192-particle deterministic capture lane, the new particle records
are 163,840 bytes rather than 1,179,648 bytes of CPU-expanded dynamic vertices
(three 48-byte vertices each): an 86.1% reduction in particle-record upload
volume before any driver-side timing claim. This is a structural byte-count
comparison, not a renderer-wide performance result.

## Boundaries and follow-up

This does not sort translucent particles, introduce indirect draws, alter
particle textures or legacy shape generation, or claim representative-map
frame-time improvement. `FR-01-T15` still owns before/after CPU/GPU timing
budgets, while `FR-01-T14` retains broader effect residency, batching, and
general transient-ring work.

No end-user documentation changed: this is a default-preserving renderer
implementation detail, and the optional diagnostic cvar is intentionally
renderer-specific.
