## Failure Observed
The Linux runner failed while compiling `src/rend_rtx/vkpt/main.c`:

```text
error: ‘VK_KHR_LINE_RASTERIZATION_EXTENSION_NAME’ undeclared here
error: unknown type name ‘VkPhysicalDeviceLineRasterizationFeaturesKHR’
```

The repository had already been updated in `src/rend_rtx/vkpt/debug.c` to use the current `EXT` line-rasterization symbols, but `src/rend_rtx/vkpt/main.c` still used the older `KHR` names in:

- the optional device-extension list
- physical-device feature probing
- device-creation feature chaining

That mismatch only surfaced on the hosted Linux runner because its Vulkan headers expose the line-rasterization extension through the `EXT` names.

## Implementation
Updated `src/rend_rtx/vkpt/main.c` so the remaining line-rasterization usage now matches the headers already handled elsewhere in the renderer:

- `VK_EXT_LINE_RASTERIZATION_EXTENSION_NAME`
- `VkPhysicalDeviceLineRasterizationFeaturesEXT`
- `VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_LINE_RASTERIZATION_FEATURES_EXT`

The optional-extension availability check was updated to use the corresponding `OPT_EXT_VK_EXT_LINE_RASTERIZATION` enum value generated from the revised extension list.

## Validation
Local verification completed:

- `meson compile -C builddir-msys2-run worr_rtx_x86_64`
- repository search confirming no remaining `VK_KHR_LINE_RASTERIZATION`, `VkPhysicalDeviceLineRasterizationFeaturesKHR`, or `VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_LINE_RASTERIZATION_FEATURES_KHR` references remain under `src/rend_rtx/vkpt/`

## Verification Limits

## Outcome
Run `23159773990` / job `67284227517` was failing because `src/rend_rtx/vkpt/main.c` had not been updated along with the earlier debug-draw line-rasterization fix. The RTX main-path extension negotiation now uses the same `EXT` symbols as the rest of the renderer, which removes the Linux CI compile failure.
