## Context
GitHub Actions run `23156641291` failed in three different places after the earlier release-packaging changes landed:

- Linux failed while compiling `rend_gl`.
- Windows failed on the same `rend_gl` translation unit and also emitted repeated first-party C++20 warnings in `sgame`.
- macOS finished building but failed during `.install/` refresh when release-pack staging tried to create `.install/worr/` on a platform where the root client executable is already named `worr`.

Run investigated:
- `https://github.com/DarkMatter-Productions/WORR/actions/runs/23156641291`

## Failures Observed

### Linux and Windows `rend_gl` failure
`src/rend_gl/main.c` had been changed to append `q_unused` after the parameter list of a function definition:

- `GL_BoxLeafs_headnode(...) q_unused`

That placement is not accepted by GCC/Clang for function definitions, so both the Linux and Windows jobs failed while compiling the OpenGL renderer.

`src/rend_gl/world.c` used the same pattern for unused helper definitions and would have remained vulnerable after the first compile failure was fixed.

### macOS `.install/` refresh failure
The refreshed staging layout tried to write the release-only asset pack directly to:

- `.install/worr/pak0.pkz`

That works on Windows because the root executable is `worr.exe`, but it is invalid on Linux/macOS because the root client executable is the file `.install/worr`. The macOS job therefore failed with `FileExistsError` when `tools/package_assets.py` attempted to create `.install/worr/`.

### Windows warning backlog
The Windows job also emitted repeated first-party C++20 warnings from `src/game/sgame/g_local.hpp`:

- constructor declarations inside `entity_iterable_t<TFilter>` still used template-id syntax (`entity_iterable_t<TFilter>(...)`)

Those warnings did not stop the job, but they were real first-party diagnostics and were worth cleaning while this run was open.

## Implementation

### 1. Moved release-pack staging to a non-conflicting source path
Release-only asset-pack staging now writes to:

- `.install/.release/worr/pak0.pkz`

instead of `.install/worr/pak0.pkz`.

This keeps `.install/` compatible with Unix-like platforms where the root executable is already `worr`, while still keeping the release-only pack inside the staging tree.

Changes:

- `tools/package_assets.py` now accepts `--output-path`
- `tools/refresh_install.py` now stages the release pack to `.release/worr/pak0.pkz`
- `tools/release/validate_stage.py` now validates that staged release-pack source path (while still tolerating the older direct layout if it exists)

### 2. Remapped the hidden staged pack into the published archive layout
`tools/package_release.py` now supports `--mapped-file source=dest`, allowing a staged file to be published into a different archive path and manifest path.

`tools/release/package_platform.py` and `tools/release/targets.py` now use that mapping so:

- staged source: `.release/worr/pak0.pkz`
- published path: `worr/pak0.pkz`

This preserves the release archive contract requested for WORR while avoiding filesystem collisions in the local staging root.

### 3. Fixed GCC-compatible unused-function annotations in `rend_gl`
Updated the OpenGL renderer helper definitions to place `q_unused` before the function declarator:

- `src/rend_gl/main.c`
- `src/rend_gl/world.c`

That restores compatibility with the GCC/Clang toolchains used in CI.

### 4. Removed the `sgame` C++20 constructor warnings
Updated `entity_iterable_t<TFilter>` in `src/game/sgame/g_local.hpp` so its constructors use standard constructor syntax:

- `entity_iterable_t()`
- `entity_iterable_t(uint32_t start)`
- `entity_iterable_t(uint32_t start, uint32_t end)`

This removes the repeated `-Wtemplate-id-cdtor` warning noise seen in the Windows job.

### 5. Reduced fallback subproject warning noise further
Extended the quiet fallback dependency options in `meson.build` to pass `-w` for fallback C/C++ subproject builds, and kept HarfBuzz's Windows-specific `_CRT_SECURE_NO_WARNINGS` define while preserving that warning suppression.

This keeps nightly logs focused on WORR regressions instead of fallback dependency chatter.
