## Current strengths

The strongest part of WORR today is that the OpenAL backend is already modern enough to host a serious spatial system. `qal.cpp` loads OpenAL 1.1, checks for `ALC_EXT_EFX`, exposes `al_hrtf`, and requests `ALC_SOFT_HRTF` when available. `al.cpp` then detects `AL_SOFT_source_spatialize`, `AL_SOFT_loop_points`, `AL_EXT_float32`, and the `AL_AIR_ABSORPTION_FACTOR` enum, and allocates per-channel OpenAL sources, direct filters, reverb filters, an effect, and an auxiliary slot. That is exactly the sort of substrate you want before doing any higher-level acoustic work. The second major strength is that WORR already has a real material path through BSP, not just guessed texture names. `csurface_t` carries both a unique texinfo ID and a `material[16]` field, which aligns with the Quake II rerelease API changes. `mtexinfo_t` stores `step_material` and `step_id`. `BSP_LoadMaterials` loads `textures/<name>.mat`, copies the material string into `c.material`, and assigns stable step IDs so identical materials share identifiers. That means the engine already has the basic data plumbing needed for material-aware acoustics without inventing a second metadata system. Occlusion is also materially better than a binary line-of-sight check. The current `S_ComputeOcclusion` function uses a direct ray plus four offset rays, scales peripheral rays by a diffraction weight, detects translucent/window cases, classifies material strings into glass, grate, soft, wood, metal, and concrete families, computes both a gain-style occlusion amount and a frequency cutoff target, then smooths both over time with separate attack and release rates. That is already a good basis for believable muffling, especially compared with classic idTech approaches. ## Current weaknesses

The single biggest weakness is that the current acoustic logic is still too **listener-local**. Reverb selection is driven by listener-side dimension probes plus a downward trace to the surface under the listener, then mapped through step IDs to a preset bucket. If there is no suitable floor hit, the code falls back to the ŌĆ£plainŌĆØ preset. That means the system classifies ŌĆ£where the player is standingŌĆØ, not ŌĆ£what acoustic space the sound occupiesŌĆØ or ŌĆ£what path joins source and listenerŌĆØ. A source in a stone corridor heard from an outdoor yard can only be approximated indirectly through send level and filtering; it cannot carry its own room identity in a principled way. ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü

Occlusion is decent, but it is not yet elegant in the BSP sense the request calls for. The current implementation uses five rays, not a true source/listener fan, hemisphere, or portal path search. It only evaluates the **first blocking hit** on each ray, so it cannot accumulate multiple layers such as grate plus concrete behind it, or glass plus interior wall. It does not use the second surface in rerelease traces, nor does it distinguish direct obstruction from ŌĆ£indirect but reachable through an apertureŌĆØ. The OpenAL EFX guide explicitly encourages repeated runtime evaluation of occlusion, obstruction, and exclusion as distinct phenomena, and modern propagation systems treat transmission and redirection as separate stages rather than as a single scalar muffling value. WORR is still collapsing all of that into one small family of traces and one low-pass target. ŅłĆfileciteŅłéturn44file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search35Ņłéturn9search5Ņłéturn10search2Ņłü

Material awareness is also only halfway modern. The present `.mat` path is the simple Quake II rerelease style: a short material ID string is loaded and associated with texinfo so footsteps and surface logic can classify materials. That is useful, but it is not an acoustic material model. Acoustic propagation wants at least coarse **banded** parameters for transmission, absorption, and scattering. Steam AudioŌĆÖs material model, for example, separates low-, mid-, and high-frequency absorption and transmission and includes scattering as a first-class property. WORR currently infers acoustic behaviour from material-name substring matches such as `glass`, `metal`, or `concrete`, then derives a single cutoff and a single weight. That is practical, but it cannot scale elegantly across richer material vocabularies, custom mods, or rerelease `.mat` extensions. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn44file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search0Ņłéturn9search5Ņłü

Outdoor detection is currently the weakest ŌĆ£environment intelligenceŌĆØ piece. The engine does have a nice trick in `AL_EstimateDimensions`: if one upward probe hits a sky surface, it inflates that axis by an extra 4096 units. That helps detect openness, but it is still a scalar average of extents, not an outdoor classifier. There is no explicit sky-visibility ratio, no areaportal-based openness measure, no distinction between canyon-like exterior spaces and exposed plains, and no source-vs-listener outdoor relation. In practice that means many important cases will collapse into the same preset bucket even though they need different late-tail, diffusion, and send behaviour. ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü

## Recommended architecture

The right target is a **three-stage acoustic resolver** that runs entirely from shipped BSP data by default, but accepts optional authored or baked augmentation.

The first stage should be a **material layer**. Keep the rerelease-compatible `.mat` ID exactly as the primary key, but introduce an internal `acoustic_material_t` lookup table that resolves each material ID into three-band transmission, three-band absorption, a scattering factor, and a few flags such as `is_sky`, `is_water`, `is_foliage`, `is_grate`, and `outdoor_hint`. If a map only supplies simple `.mat` IDs, WORR should infer acoustic values from a built-in default map. If future `.mat` parsing is extended, those band values can be overridden in data without breaking rerelease compatibility. This mirrors how modern propagation systems use a lightweight semantic material layer instead of hard-coding substring tests as the final authority. ŅłĆfileciteŅłéturn27file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn31file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search0Ņłéturn9search5Ņłü

The second stage should be a **BSP acoustic topology layer**. On map load, derive an `audio_leaf_t` or `audio_area_t` cache from BSP leafs and areaportals. For each static region, precompute a dominant-material histogram, a sky visibility score, vertical openness, approximate room scale, portal neighbours, and whether the region behaves like interior, semi-open, or exterior space. This requires no offline compiler: the BSP already contains leafs, areas, areaportals, visibility data, and sky flags. The key design change is simply to compute this once and retain it as acoustic metadata instead of recomputing ad hoc only from the listener every frame. ŅłĆfileciteŅłéturn27file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü

The third stage should be **path resolution**. For every audible source, WORR should resolve one of four path classes: same-space direct, direct-through-material, indirect-through-portal, or acoustically unreachable. Direct paths can continue using rays, but they should gather more than one blocking surface where possible and combine materials in bands instead of using only the first hit. If the direct path is blocked, the resolver should consult the area/portal graph and attempt a lightweight portal route. This does not need full wave simulation. Even a simple shortest-path search over areaportals, weighted by aperture size, bend angle, and cumulative material loss, will produce far better ŌĆ£around the cornerŌĆØ behaviour than a pure direct trace. That recommendation is strongly aligned with the distinction that Project Acoustics makes between basic occlusion and true redirection/portaling, and with the EFX guideŌĆÖs separation of obstruction, occlusion, and exclusion. ŅłĆciteŅłéturn10search1Ņłéturn10search2Ņłéturn9search35Ņłü

Once those three stages exist, the OpenAL backend becomes much simpler to reason about. The direct path drives `AL_GAIN`, `AL_DIRECT_FILTER`, and optional air absorption. The listener-room descriptor drives the global EFX slot. The source-to-listener path class drives reverb send amount, reverb-send filtering, and optional early-reflection behaviour. In other words: **OpenAL stays the DSP engine; BSP becomes the acoustic reasoning engine**. That is a cleaner separation than the current mixture of per-source traces, listener-floor probes, and zone toggles. ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search7Ņłéturn9search35Ņłü

Optional compiled data should remain exactly that: optional. A lightweight `.aud` sidecar could store precomputed leaf descriptors, portal apertures, and portal-route weights, improving large-map performance and around-corner quality. That is consistent with broader industry practice: Steam Audio exposes both real-time simulation and more expensive reflection options, while Project Acoustics explicitly shifts costly wave-style propagation into a bake step for better diffraction and portaling. The crucial point for WORR is that the baseline runtime should still sound good **without** such a file. The sidecar should improve quality and cost, not unlock basic plausibility. ## Implementation roadmap

The first pass should be a **consolidation pass**, not a feature pass. Turn the current advanced features into sane defaults, decouple reverb send routing from `al_eax`, keep inverse distance as the default model, set reverb and air absorption on by default, and strongly consider moving HRTF to ŌĆ£default/autodetectŌĆØ rather than ŌĆ£offŌĆØ for the default cvar state. Also ship a compiled-in fallback reverb-environment table so audio quality does not collapse if `sound/default.environments` is absent. This pass is mostly low-risk engineering and tuning, and it would immediately move WORR closer to the userŌĆÖs goal of ŌĆ£modern, high-quality sound by defaultŌĆØ. ŅłĆfileciteŅłéturn37file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn44file0ŅłéL1-L1Ņłü

The second pass should replace the current substring occlusion classifier with a **real acoustic-material resolver**. Keep the existing material families and cutoff defaults as the initial preset table, but convert them into banded coefficients and make `.mat` IDs the key. Add a tiny built-in mapping for common rerelease materials and multiplayer staples so maps with only simple `.mat` files still behave sensibly. The runtime path then becomes: resolve material ID ŌåÆ acoustic coefficients ŌåÆ combine over hits/portals ŌåÆ produce direct gain, per-band low-pass, and send colour. That upgrade preserves backward compatibility while making the system extensible and much cleaner. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn44file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search0Ņłéturn9search5Ņłü

The third pass should add **BSP acoustic regions**. Build a map-load cache keyed by leaf or area that stores openness, sky ratio, dominant material groups, and portal neighbours. Replace the current listener-floor-driven preset choice with a resolver that classifies both source space and listener space. Reverb preset selection should use a weighted decision from region type, openness, and material composition, not just the material underfoot. Outdoor classification should be explicit and multi-factor: sky exposure, portal connectivity, vertical openness, and enclosing surface proportion. That one step will solve most of the current ŌĆ£indoor/outdoor misclassificationŌĆØ problems. ŅłĆfileciteŅłéturn27file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü

The fourth pass should implement **portal-aware propagation**. Start modestly: direct multi-ray path first, then a one- or two-portal fallback through areaportals or explicitly marked openings. Each leg accumulates distance, aperture penalty, bend penalty, and material transmission. This is where WORR should finally use BSP structure elegantly. It will also eliminate the most visibly gamey cases: a thin post fully ŌĆ£occludingŌĆØ a sound, or a sound beyond a doorway being treated as if it were simply behind a monolithic wall. Project AcousticsŌĆÖ research summary is a good conceptual reference here because it explicitly shows why redirection through doors and diffraction around corners matters perceptually. ŅłĆciteŅłéturn10search1Ņłéturn10search2Ņłéturn9search35Ņłü

The fifth pass should revise reverb into a **two-identity model**: listener-room identity and source-room path state. The EFX slot remains globally tied to the listener environment, because that is what the ear is ŌĆ£insideŌĆØ, but each source gets send gain and send filtering derived from its own resolved path. Same-room sources stay dry and clear. Adjacent-room sources get less direct energy and more filtered send. Exterior-to-interior and interior-to-exterior transitions get distinct send behaviour instead of sharing one generic occlusion curve. If the implementation budget allows it, a second auxiliary slot for early reflections or portal slap may be worthwhile; if not, one slot plus path-aware send filtering is still a strong improvement. Steam AudioŌĆÖs split between direct-path controls, transmission, and optional reflections is a useful design analogue here. ŅłĆfileciteŅłéturn43file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search4Ņłéturn9search5Ņłéturn9search8Ņłü

The sixth pass should be optional enhancement. Support `client_env_sound` and `env_sound` as authored overrides, continue consuming EAX JSON profiles, and add an optional `.aud` sidecar for maps that want finer portal or region hints. But keep the philosophy strict: authored files refine, they do not rescue. That is how you satisfy the userŌĆÖs requirement that spatial audio should work to a good standard without special scripts or compiled data, while still leaving headroom for bespoke single-player maps or showcase environments. One limitation is that I could verify Q2RTXPerimentalŌĆÖs role directly only through WORRŌĆÖs own `al.cpp` comment and the structures already present in WORR, not through a full line-by-line audit of that selected external repository.
